# 执行纪律工具箱

两个核心模式，解决同一类问题：**建了白建、跑了白跑**。

---

## 模式一：闭环铁律（新建任何工件后必做）

### 问题
创建了文件/脚本/cron/配置，但没有建立调用链路——没人知道它什么时候被读取，它就变成了孤儿文件。

### 三问自检清单
建完任何工件后，必须回答这三个问题（答不出=未完成，必须补完再交差）：

1. **谁会在什么场景下读它？** → 写入调用规则到触发点（SOUL.md 做事方式 / ARCHITECTURE.md 对应章节 / Skill 的触发条件）
2. **调用规则写在哪个文件里了？** → 确认规则已实际写入
3. **那个文件里有没有指向它的链接？** → 父文档有反向引用

### 交叉引用模式
推荐的引用方式——双向链路：
```
父文档 → "涉及XX操作前，先读 目标文件"
目标文件 → "本文件的调用规则写在 父文档 XX条目"
```

### 反面案例
- CRON_JOBS.md 建好了，但 SOUL.md 没写「Cron操作前先读CRON_JOBS.md」→ 文件存在但永远没人读
- ARCHITECTURE.md 建好了，但没有从任何地方引用 → 孤儿文档
- feishu_output_rules.md 建好了，但 SOUL.md 沟通风格没指向它 → 规则被遗忘

## 模式二：任务生命周期管理（重做/启动任务前必做）

### 问题
任务A有问题→开了任务B→任务A还在跑→两个并行空跑浪费资源。或：旧任务已跑过但不知道→新任务重复执行→撞API限流。

### 启动前四查
在启动任何新任务（特别是重做已有任务）之前：

1. **查旧任务**：`cronjob list`，同名/同类旧cron是否还在跑 → 问老大要不要停
2. **查进度**：读 progress 文件，确认已完成量 → 不要从零开始
3. **查配额**：如果是批量API操作，查剩余配额 → 撞429就亏了
4. **查依赖**：数据源是否就绪、前置任务是否完成

### 完成后必做
- cron任务完成后：必须 `remove` 或 `pause`，不能让已完成的任务无限循环空跑
- 一次性任务：设 `repeat=1`，不要设 `repeat=forever`
- 进度文件：每批完成后更新，下次启动时读取

### ⚠️ Cron Prompt 铁律：绝不用模糊引用替代具体命令

**问题**：更新 cron prompt 时，把原来完整的命令步骤压缩成"同之前逻辑"、"和之前一样"之类的模糊引用。cron agent **没有上下文记忆**——每次运行都是全新 session，它不知道"之前"是什么。

**铁律**：
1. cron prompt 里的每个数据获取步骤必须是**可独立执行的完整命令**
2. 禁止"同之前"、"和之前一样"、"参考上面"——agent 不知道"之前"是哪次
3. 更新 cron prompt 时，如果只改了部分内容，**未改部分也必须保留完整命令**
4. 加强调语：`必须用 terminal 实际执行，不能猜、不能编造文件路径`

## 模式三：Memory维护（周期性自检）

### 问题
Memory条目堆积、过时、重复、或包含不该存的内容（如密码、临时状态）。

### 自检清单
定期（每次整理memory时）逐条检查：

1. **这条还准确吗？** → 过时信息比没信息更危险（会基于错误前提做决策）
2. **这条有其他地方存了吗？** → 已迁入文件/TencentDB/user-profile的条目应删除
3. **这条是长期的还是临时的？** → 临时状态（进度、配额、当日任务）不存memory
4. **这条包含敏感信息吗？** → 密码、token、私钥绝不存memory（每轮注入system prompt）
5. **这条有块标签吗？** → 【系统】【基建】【索引】【项目】【规则】等，方便扫描

### Memory定位
Memory = **索引目录** + **运行时状态**，不是内容仓库。
- 内容在文件（SOUL.md / ARCHITECTURE.md / CRON_JOBS.md）
- 用户信息在 TencentDB user-persona / USER.md
- 临时状态在 progress 文件 / CRON_JOBS.md
- Memory只存：系统身份、基础设施状态、文件索引、长期规则、活跃项目关键路径

## 模式四：文件设计规范（新建/重构MD文件时必做）

### 问题
文件写了但没人读——要么太大（550行没人会每次读），要么定位模糊（什么都往里塞），要么没有设计约束（内容无限膨胀）。

### 文件价值公式
> **文件的价值 = 有人读 × 有人用**
> 如果调用频率为0，说明文件太大/触发太模糊，需要重新设计。

### 设计方法论约束（每类文件置首行）
新建MD文件时，在文件首行用HTML注释写入设计方法论，作为硬性约束：

| 文件类型 | 方法论 | 约束 |
|---------|--------|------|
| 行为规则（SOUL.md） | 系统思维 + 约束驱动 | 规则间不能打架，2200字符上限 |
| 架构文档（ARCHITECTURE.md） | 系统思维 + 概念澄清 | 术语首次定义，组件依赖关系准确 |
| 运营清单（CRON_JOBS.md） | 决策理论 + 风险分层 | 每个任务标注频率/风险/失败策略 |
| 用户档案（USER.md） | 分类范畴分析 + 重构问题 | 按更新频率分层 |
| 格式规则（feishu_rules.md） | 约束驱动 + 反例检验 | 规则可判断，边界有明确答案 |
| 运行时索引（MEMORY.md） | 逆向思考 + 满意化 | 删掉会出错的才值得放 |

## 模式五：进度回报（长时间执行必做）

### 问题
连续跑多个工具调用（写文件→启动服务→测试→调试），每个等几秒到几十秒，串起来十几分钟无任何输出给用户。用户看到的是"session still working"，不知道进展、不知道卡在哪、不知道还要多久。

### 规则
**每做1步操作，最少5分钟内给老大1条状态回报。** 绝不闷头干活超过5分钟不吭声。

### 实操方式
1. **开始前**：告诉老大"我要干这几步"（1句话概要，不啰嗦）
2. **每步完成后**：简短回报"第X步完成，当前状态是XXX"
3. **遇到阻塞时**：立刻说"卡在XXX，原因是YYY，正在处理"
4. **不要等全部做完再汇报** — 分段回报 > 一口气交差
